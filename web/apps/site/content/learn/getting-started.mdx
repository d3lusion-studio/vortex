## Requirements

| Tool | Minimum | Notes |
| --- | --- | --- |
| CMake | 3.23 | Presets v6 |
| Ninja | any | Default generator |
| C++ compiler | GCC 13 / Clang 17 / MSVC 19.38 | C++20 |
| Vulkan SDK | 1.3 | Only for the Vulkan backend |

## Build

```bash
git clone https://github.com/d3lusion-studio/vortex
cd vortex
cmake --preset release
cmake --build --preset release
```

Then run an example:

```bash
./build/release/examples/sprites
```

<Callout type="info">
  The renderer backend is chosen at runtime by `VORTEX_RHI_API` (`vulkan` or `webgpu`), falling
  back to the `VORTEX_RHI_BACKEND` CMake cache value. On Wayland the examples currently force
  X11/XWayland until the native Vulkan swapchain path lands.
</Callout>

## Your first scene

`App` owns the window, the device, the registry and the loop. Ask it for a subsystem and it is
created lazily for the active scene — there is no `PhysicsWorld` to construct and no `step()` to
remember to call.

```cpp
#include "vortex/app/app.hpp"
#include "vortex/physics/components.hpp"

using namespace vortex;

int main() {
    app::App app{{.title = "My game", .width = 1280, .height = 720}};

    const ecs::Entity player = app.scene().spawn();
    app.registry().get<ecs::Transform2D>(player).position = {0.0f, 4.0f};
    app.registry().emplace<ecs::SpriteComp>(player, ecs::SpriteComp{
        .texture = app.whiteTexture(),
        .color   = Color::white(),
        .size    = {1.0f, 1.0f},
    });
    app.registry().emplace<physics::RigidBody2D>(player, physics::RigidBody2D{
        .restitution = 0.4f,
    });
    app.registry().emplace<physics::BoxCollider2D>(player, physics::BoxCollider2D{
        .halfExtents = {0.5f, 0.5f},
    });

    return app.run();
}
```

Link against the engine target:

```cmake
find_package(Vortex REQUIRED)
add_executable(my_game main.cpp)
target_link_libraries(my_game PRIVATE Vortex::vortex)
```
